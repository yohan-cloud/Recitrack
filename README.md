# ReciTrack v1.0

Role-based classroom recitation tracking web app for teachers and students.

## Features

- Login-only authentication with JWT access tokens, refresh tokens, and bcrypt password hashes
- Forgot-password reset codes sent by email using SMTP
- Production hardening for secure refresh cookies, stricter login rate limits, strong password validation, and production secret checks
- Teacher and student roles with protected routes
- Teacher setup flow: sections, classes, students, recitations
- Student accounts generated by teachers with temporary passwords
- First-login password change flow for students
- Teacher dashboard scoped by selected class and section
- Class details page with search, random picker, compact recitation counters, edit count modal, and daily summary
- Teacher announcements scoped to a selected class and section, with type labels, optional expiration, student unread badges, and teacher read/unread counts
- Student dashboard and recitation history scoped to the logged-in student
- Teacher analytics and reports scoped to a selected class
- Teacher settings persisted in PostgreSQL for profile, school, theme, and class preferences
- PostgreSQL schema managed by Prisma

## Tech Stack

- Frontend: React, TypeScript, Vite, Tailwind CSS
- Backend: Node.js, Express, TypeScript
- ORM: Prisma 7 with PostgreSQL adapter
- Database: PostgreSQL
- Auth: JWT, refresh token cookie, bcrypt
- Validation: Zod
- Security: Helmet, CORS, express-rate-limit
- Production reliability: request logging, API 404 handling, centralized error responses

## Production Security Checklist

See [DEPLOYMENT.md](./DEPLOYMENT.md) for the deploy checklist.

Before deploying, do not use the local demo `.env` values.

Backend production `.env` example:

```env
NODE_ENV=production
PORT=4000
DATABASE_URL="postgresql://APP_DB_USER:LONG_RANDOM_PASSWORD@HOST:5432/recitrack?sslmode=require"
JWT_ACCESS_SECRET="generate-a-unique-random-string-at-least-32-characters"
JWT_REFRESH_SECRET="generate-a-different-random-string-at-least-32-characters"
JWT_ACCESS_EXPIRES_IN="15m"
JWT_REFRESH_EXPIRES_IN="7d"
FRONTEND_URL="https://your-recitrack-frontend.example.com"
REFRESH_COOKIE_SAME_SITE="lax"
REFRESH_COOKIE_SECURE="true"
SMTP_HOST="smtp.example.com"
SMTP_PORT="587"
SMTP_USER="smtp-user"
SMTP_PASS="smtp-password"
SMTP_FROM="ReciTrack <no-reply@example.com>"
```

Frontend production `.env` example:

```env
VITE_API_URL="https://your-recitrack-api.example.com/api"
```

Production requirements:

- Use HTTPS for both frontend and backend.
- Use a hosted PostgreSQL database with backups enabled.
- Do not use `postgres/postgres` or any default database password.
- Use a restricted application database user, not the PostgreSQL superuser.
- Generate different long random values for `JWT_ACCESS_SECRET` and `JWT_REFRESH_SECRET`.
- Keep `.env` files out of Git.
- Set `FRONTEND_URL` to the exact deployed frontend origin. CORS is locked to this origin in production.
- Refresh token cookies are `httpOnly` and `secure` in production, so HTTPS is required.
- If frontend and backend are on different sites, set `REFRESH_COOKIE_SAME_SITE="none"` and keep `REFRESH_COOKIE_SECURE="true"`.
- `/api/auth/login` has a stricter production rate limit.
- Password changes require at least 10 characters with uppercase, lowercase, number, and symbol.
- Password reset uses a 6-digit code sent to the email saved on the teacher or student profile. SMTP settings are required in production.
- API errors use centralized handling. Production responses hide internal stack traces; development responses include stack details for debugging.

## Setup

Install dependencies:

```bash
cd backend
npm install

cd ../frontend
npm install
```

Create a PostgreSQL database named `recitrack`.

Run Prisma:

```bash
cd backend
npm run prisma:generate
npm run prisma:migrate -- --name init
npm run prisma:seed
```

For production deploys, use migrations without the dev prompt:

```bash
cd backend
npx prisma migrate deploy
npm run prisma:generate
```

Create the first production teacher account without seeding demo data:

```bash
cd backend
TEACHER_USERNAME="teacher.username" \
TEACHER_PASSWORD="StrongPassword123!" \
TEACHER_FIRST_NAME="First" \
TEACHER_LAST_NAME="Last" \
TEACHER_EMAIL="teacher@email.com" \
npm run create:teacher
```

On Windows PowerShell:

```powershell
cd backend
$env:TEACHER_USERNAME="teacher.username"
$env:TEACHER_PASSWORD="StrongPassword123!"
$env:TEACHER_FIRST_NAME="First"
$env:TEACHER_LAST_NAME="Last"
$env:TEACHER_EMAIL="teacher@email.com"
npm run create:teacher
```

The password must have at least 10 characters with uppercase, lowercase, number, and symbol.

Start backend:

```bash
cd backend
npm run dev
```

Start frontend:

```bash
cd frontend
npm run dev -- --host 127.0.0.1
```

Open:

```text
http://127.0.0.1:5173/login
```

## PostgreSQL Troubleshooting

If Prisma says authentication failed, check `backend/.env`:

```env
DATABASE_URL="postgresql://postgres:YOUR_PASSWORD@localhost:5432/recitrack"
```

For this local setup, the PostgreSQL `postgres` password was reset to:

```text
postgres
```

If `npm run prisma:migrate` hangs, a previous Prisma migration process may still be running. Stop the old Node/Prisma process, then run:

```bash
npm run prisma:migrate -- --name init
```

If the browser shows a blank page, check the Vite terminal logs. A blank page usually means a frontend runtime error, not a database issue.

## Notes

- Analytics uses `recharts`, so Vite may warn that the bundle is larger than 500 kB. The build still works. A later optimization can lazy-load Analytics and Reports.
- Reports are intentionally class-scoped by default, matching the app rule that teacher stats should not be global unless explicitly selected.
